# 自适应工作流引擎 (Adaptive Workflow Engine)

**Feature**: `001-adaptive-workflow-engine`
**Created**: 2026-07-06
**Status**: Implemented (2026-07-06)

---

## 问题陈述

用户需要经常在 Kimi Code CLI 中执行重复的多步骤开发流程（如：审查代码 → 修改 → 运行测试 → 更新文档），每次都需要新建 session、逐条手动输入指令、人工判断回复是否正确。当前 `run_flow` 工具已能分步执行预定义步骤，但存在三个核心局限：

1. **无法学习**：每次执行相同流程都需重新描述步骤，不能从历史 session 中提取模板
2. **无法自适应**：步骤固定，不能根据任务 session 的回复微调下一条指令的措辞和细节
3. **全量轮询**：通过 poll_session 轮询状态消耗大量 token，且无可视化监管界面

## 解决方案

构建一个**自适应工作流引擎**——统筹 session 通过口头描述或历史 session 分析学习用户的重复流程，将流程提炼为可复用模板。执行时自动创建任务 session 并启用 auto 模式，逐条下发指令，通过 API 回调实时感知任务 session 状态（不轮询），在回复模糊时读取思考链确认意图，阻塞时自主处理（如安装依赖、调整步骤细节），仅将关键决策点和完成状态汇报给用户。Web 页面提供轻量可视化监管。

---

## 用户故事

1. **作为项目负责人**，我想要口头描述几次我的重复开发流程后系统能自动记住，以便下次一键启动，不用再每次手动输入步骤
2. **作为项目负责人**，我想要引擎能根据任务 session 的实际回复自动调整下一步指令（如回复"文件不在预期位置"时自动告知 session 先检查目录），而不是机械执行固定步骤导致卡住
3. **作为项目负责人**，我想要在任务 session 执行过程中通过 Web 页面看到实时进度，而不是反复调用轮询工具消耗 token
4. **作为项目负责人**，我想要在任务 session 严重阻塞（如权限不足、外部服务不可用）时才被通知决策，日常小问题（如缺依赖）由统筹 session 自主处理
5. **作为项目负责人**，我想要任务 session 一轮标准流程完成后收到简洁汇报，决定是继续下一轮还是关闭 session 开启新任务
6. **作为平台运营者**，我想要工作流模板以配置文件形式持久化，支持版本管理和跨 session 复用

## 功能需求

### FR-1：模板学习

- FR-1.1：用户可通过口头描述定义工作流，系统从中提取步骤序列、项目规范引用和预期产出
- FR-1.2：用户可指定一个已完成 session ID，系统分析该 session 的输入输出记录（`list_io_records`），提取用户的指令模式为模板
- FR-1.3：学习后的模板保存为结构化配置文件（YAML），包含步骤列表、规范文档路径、超时设置、恢复策略
- FR-1.4：模板支持手动编辑和版本管理

### FR-2：自适应步骤驱动

- FR-2.1：执行模板时，按步骤顺序逐条通过 `execute_prompt(wait=true, auto_mode=true)` 下发指令到任务 session
- FR-2.2：每步执行后分析任务 session 的回复，若回复明确推进则进入下一步
- FR-2.3：若回复模糊或不明确，自动调用 `read_session_log(include_thinking=true)` 读取思考链内容确认意图
- FR-2.4：若回复指示遇到阻塞（如缺少依赖、路径错误），统筹 session 自主判断处理方式：
  - 环境问题 → 告知任务 session 自行安装/配置
  - 规范理解偏差 → 重新下发带更明确规范的指令
  - 不可自动处理的严重阻塞 → 暂停并通知用户
- FR-2.5：自适应调整不改变原步骤的核心目标（非降级处理），仅调整执行路径

### FR-3：状态感知（低频轮询，非全量日志）

- FR-3.1：由于 Kimi Server 不提供原生状态变更回调，采用低频轮询策略：每 3-5 秒查询 `GET /api/v1/sessions/{id}/status` 端点（仅返回 `status`/`model`/`thinking_level` 字段，不拉全量日志），token 消耗远低于 `poll_session`
- FR-3.2：不使用 `poll_session` 轮询（后者每次读取 wire.jsonl 尾部），改为仅查轻量 `status` 端点
- FR-3.3：若 `status` 端点不可用，降级为 `poll_session` 备用

### FR-4：阻塞处理预案

- FR-4.1：预定义常见阻塞类型及处理策略表：
  - 工具审批等待 → 自动审批（auto_mode）
  - 依赖缺失 → 告知 session 安装
  - 文件不存在 → 告知 session 检查路径
  - 权限不足 → 暂停，通知用户
  - 超时 → 重试一次，再超时则暂停
- FR-4.2：用户可自定义阻塞处理策略

### FR-5：Web 监管页面

- FR-5.1：轻量 HTML 页面，通过 WebSocket 接收统筹 session 推送的实时状态
- FR-5.2：显示当前执行模板名称、当前步骤、任务 session ID、已完成步骤数、最近回复摘要
- FR-5.3：显示阻塞警报（红色高亮）及建议操作
- FR-5.4：页面不依赖外部服务，由 tunnel 的 HTTP server 直接托管

### FR-6：流程控制

- FR-6.1：一轮标准流程（模板所有步骤执行完毕）后，向用户汇报完成状态
- FR-6.2：用户可选择：继续下一轮（重新执行模板）、通过 `continue_workflow(decision="manual", instruction=...)` 临时覆盖当前步骤继续（不修改模板文件）、关闭任务 session 并创建新 session 执行下一任务
- FR-6.3：支持中断执行——用户随时可暂停或终止工作流

## 成功标准

| # | 标准 | 指标 |
|---|------|------|
| SC-1 | 用户口头描述 3 次以内相同流程后，系统能生成可用模板 | 生成模板的步骤完整度 ≥ 80% |
| SC-2 | 模板执行过程中的人工介入次数显著减少 | 相比手动操作减少 70% 以上的交互轮次 |
| SC-3 | Web 页面实时展示进度，延迟不超过 3 秒 | 状态变更到 Web 显示的延迟 |
| SC-4 | 常见阻塞（依赖缺失、路径错误）自动处理成功率 ≥ 80% | 无需用户介入的阻塞处理占比 |
| SC-5 | 模板执行整体 token 消耗不高于手动操作 | 含 Web 页面推送的总 token 量 |
| SC-6 | 用户从启动模板到查看完成汇报的操作步骤 ≤ 3 步 | 1.选择模板 2.确认启动 3.查看结果 |

## 核心实体

| 实体 | 描述 |
|------|------|
| **WorkflowTemplate** | 工作流模板：步骤列表 + 项目规范路径 + 阻塞策略 + 超时配置。持久化为 YAML 文件 |
| **WorkflowExecution** | 一次工作流执行实例：关联模板、任务 session ID、当前步骤、完成状态、阻塞记录 |
| **StepResult** | 单步执行结果：步骤序号、下发指令、session 回复（含思考链摘要）、是否需自适应调整 |
| **BlockageEvent** | 阻塞事件：类型、上下文、处理策略、是否需要用户决策 |

---

## 不在范围内

- 不替代 `run_flow`——在其架构之上增强，`run_flow` 继续服务于简单顺序流
- 不做跨项目模板迁移（模板与项目规范绑定）
- 不做自然语言模板查询（用户通过 CLI 选择模板）
- 不做多任务 session 并行调度（每次一个任务 session）
- Web 页面不做用户认证——由 tunnel 本地端口提供，信任本机访问

---

## 假设

- Kimi Server API 的状态变更回调机制可用（或可通过 WebSocket 事件近似实现）
- 用户的工作流可在单 session 内完成一轮（多轮可通过重复执行模板实现）
- 项目规范文档存在于项目路径下，统筹 session 可直接读取
- `wire.jsonl` 中的 `content.part(think)` 字段包含可判读的思考内容
- 用户允许统筹 session 自动审批工具调用（auto_mode）
