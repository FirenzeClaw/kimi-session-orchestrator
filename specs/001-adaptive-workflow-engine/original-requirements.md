# 原始需求：自适应工作流引擎

> 来源：2026-07-06 用户对话中逐步明确的原始需求描述
> 版本：v1.0

---

## 用户原始描述

我的设想是，该MCP工具能够实现，一个统筹session，能够根据用户指示学习用户的重复工作指令，在需求的项目路径启动任务session并启用为auto模式，然后一步一步的告知session下一步并根据session的每轮回复自动微调下一步（非降级处理）输入的指令完成一轮任务，一轮标准任务完成后汇报给用户session的完成状态并等待用户决策，确认用户是否关闭任务session并启用新任务session再次执行流程，直到用户认为任务session已经达到目标完成度。

---

## 逐轮提炼

### 第 1 轮：交互方式
> "不同于创建session后一股脑将需要他做的事都输入然后任由他执行，需要分步骤，第一步输入后等待他的回复，然后输入第二步，再等待回复，直到我给出的标准步骤流程都实施完成"

**→ 结论**：不是一次下发全部任务，而是逐条 wait → response → next。

### 第 2 轮：自适应微调
> "不使用修改、创建等描述，而是告知session遵循项目的指导规范处理...若存在session有依赖环境等缺失需求，允许任务session安装或者统筹session替他配置好后告知，且统筹session需要理解项目的规范，保证统筹时不出现偏差"

**→ 结论**：微调 = 向项目规范对齐。统筹 session 需理解规范文档。环境问题可分两种：轻量由任务 session 自行安装，复杂由统筹预配置。

### 第 3 轮：可见性 vs 自主性
> "创建一个web页面可监管即可，节约token，只在任务session遇到严重阻塞统筹session无法提供帮助时等待用户决策，其他情况下只在任务session完成任务后给用户反馈"

**→ 结论**：Web 页面轻量监管。用户只在两种情况下参与：严重阻塞 + 任务完成汇报。其余全程自动。

### 第 4 轮：交互方式升级
> "不要使用轮询，而是直接通过API的回调交互，并且当任务session卡住时要有处理预案，任务session回复不明确时统筹session能够通过读取特定位置的思考链确认"

**→ 结论**：使用 API 回调/轻量状态查询（非 poll_session）。阻塞要有预定义处理策略。回复模糊时读 wire.jsonl 的 think 部分。

### 第 5 轮：学习方式
> "既可以口头描述，又可以根据用户给出的session ID读取用户的输入"

**→ 结论**：两种学习来源——口头描述 + 从已完成 session 提取 prompt 模式。

## 最终需求摘要

| 维度 | 描述 |
|------|------|
| **统筹模式** | 分步下发指令，每步等待回复后再进入下一步 |
| **学习能力** | 口头描述 + 历史 session 提取 → 生成可复用模板 |
| **自适应** | 基于项目规范对齐，非降级；模糊时读思考链确认 |
| **阻塞处理** | 预定义策略表，自动处理轻量阻塞，严重阻塞暂停等用户 |
| **状态感知** | 轻量 API 状态查询（非 poll_session），节约 token |
| **Web 监管** | 轻量页面，实时进度，阻塞警报 |
| **用户参与** | 仅两种场景：严重阻塞决策 + 任务完成汇报 |
| **流程控制** | 完成后可选继续/新建/关闭 session |
