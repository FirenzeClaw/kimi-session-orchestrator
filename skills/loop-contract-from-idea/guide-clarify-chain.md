# Guide — 5 轮追问链

加载条件：在 `loop-contract-from-idea` Phase 1 执行。仅当输入是一句话需求或 `/loop-contract-new` 且尚未确认存在完整文档集时加载。

## 核心规则

1. 五轮必须按 Q1 → Q5 顺序执行。
2. 每轮都要记录：用户原答、是否充分、追问内容、最终可用结论。
3. 不充分回答不能直接跳过；必须使用本 guide 的追问策略至少追问一次。
4. Q5 发现已有文档时，立即升级到 `loop-contract-from-docs` 路径，不再继续 from-idea 推断。
5. Auto permission mode 下只用纯文本提问，不调用 `AskUserQuestion`。

## 追问记录格式

```markdown
### Q<N> <主题>
- 用户原答: <原文摘要>
- 充分性: 充分 | 不充分但已追问 | 升级 from-docs
- 追问: <若有>
- 可用结论: <进入后续 Phase 的事实>
```

## Q1 — 最终目标

问题：谁使用这个能力？要完成什么动作？为什么这件事重要？

退出标准：同时得到三类信息：

- 使用者或受影响角色。
- 目标动作或最终产出。
- 价值或失败代价。

不充分回答的追问策略：

- 若只有功能名或抽象目标，追问：“能否描述一个具体的使用场景？”
- 若缺少使用者，追问：“这个能力主要给谁使用或影响谁？”
- 若缺少重要性，追问：“如果这个目标没完成，会造成什么具体问题？”

允许记录“不充分但已追问”的条件：用户追问后仍拒绝补充或明确表示不知道。此时可用结论必须标注“Q1 信息不足”，后续 AC 只能写保守、可验证的目标。

## Q2 — 平台/环境约束

问题：运行平台和环境约束是什么？包括 OS、浏览器、嵌入式环境、部署形态、性能基线、离线/联网要求。

退出标准：至少明确以下之一：

- 目标平台或默认平台。
- 必须兼容或明确不支持的环境。
- 性能、资源或部署基线。

不充分回答的追问策略：

- 用户说“不限”或“都行”时，追问：“默认Web应用，确认？”
- 用户只说“电脑端”时，追问：“是否需要限定 Windows/macOS/Linux 或浏览器版本？”
- 用户提到性能敏感但无数值时，追问：“是否有可接受的响应时间、内存或并发基线？”

允许记录“不充分但已追问”的条件：用户确认无特殊约束。此时可用结论写“默认 Web 应用，无额外平台约束”。

## Q3 — 技术栈

问题：语言、框架、数据库、现有系统或集成依赖是什么？

退出标准：满足任一条件即可：

- 明确语言/框架/数据库或现有项目路径。
- 用户明确表示“不限”。
- 用户要求由 PM 根据项目现状选择。

不充分回答的追问策略：

- 用户说“不知道”时，追问：“是否允许 PM 根据当前项目技术栈推断？”
- 用户只给语言不提框架时，追问：“框架或数据库是否已有指定？”
- 用户明确“不限”时，不再追问，记录为“技术栈不限，可按项目现状选择”。

允许跳过细化的条件：用户明确“不限”。跳过不等于空白；记录“技术栈不限”。

## Q4 — 方向约束

问题：明确不做什么？不碰哪些模块？有哪些边界不能被 PM 或 task session 降级、绕开、替换？

退出标准：至少得到一类方向基线：

- 明确不做的范围。
- 不可修改或不可触碰的模块/接口/数据。
- 不可接受的降级方式。
- 用户明确“暂无方向约束”。

不充分回答的追问策略：

- 用户回答“没有”时，追问：“是否允许 PM 保持最小改动，只实现目标必要范围，不做额外重构？”
- 用户只描述要做什么时，追问：“有没有明确不希望改动的模块、接口或行为？”
- 用户给出模糊边界时，追问：“请把这个边界改写成一句可检查的‘不做/不碰’规则。”

允许记录“不充分但已追问”的条件：用户确认无边界。此时可用结论写“默认方向约束：最小改动，不做目标外重构，不缩减已确认目标”。

## Q5 — 是否已有相关文档

问题：是否已有 SPEC、PLAN、PRD、TASK、设计文档、issue、README 章节或其他相关文档？如有，请提供路径。

退出标准：明确有或没有文档：

- 有文档：拿到至少一个路径或文档位置。
- 无文档：用户明确没有，继续 from-idea。

不充分回答的追问策略：

- 用户说“可能有”时，追问：“请提供你知道的路径；如果暂时找不到，我会继续 from-idea 并标注无文档输入。”
- 用户给出目录不具体时，追问：“是否可以把这个目录视为 from-docs 输入并进入文档侦察？”
- 用户粘贴文档摘要但无路径时，追问：“是否有原始文档路径？有路径则升级 from-docs；没有则作为 user-input 继续。”

升级路径：

1. 若用户提供任何可读文档路径，停止 from-idea Phase 2。
2. 切换到 `loop-contract-from-docs`，把 Q1-Q4 记录作为补充上下文交给 from-docs。
3. Contract 的 `generated_by` 应使用 from-docs 路径规则，`source_docs` 列出文档路径。

继续 from-idea 的条件：用户明确无文档，或追问后仍无可用路径。

## Phase 1 完成清单

| # | 检查项 | 必须满足 |
|---|--------|----------|
| 1 | Q1 | 有使用者/动作/重要性，或记录“不充分但已追问” |
| 2 | Q2 | 有平台默认或环境约束，或确认无特殊约束 |
| 3 | Q3 | 有技术栈、允许 PM 推断，或明确“不限” |
| 4 | Q4 | 有方向约束，或确认默认最小改动边界 |
| 5 | Q5 | 已判定 from-idea 继续或升级 from-docs |

任一项缺失，禁止进入基线锁定。
