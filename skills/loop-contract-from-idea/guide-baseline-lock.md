# Guide — 基线锁定协议

加载条件：在 `loop-contract-from-idea` Phase 2 执行。仅当 Phase 1 的 Q1-Q5 已完成且 Q5 未升级到 from-docs 时加载。

## 核心规则

从一句话需求得到的 AC 和方向基线都是 PM 推断，不是事实。只有用户确认后，才能写入 Contract；写入后不得自行修改。

## Step 1 — AC 推断

从 Q1-Q4 回答中提取可验证条件：

| 来源 | 提取规则 | 示例 |
|------|----------|------|
| Q1 目标动作 | 转成最终可观察结果 | “用户能导入文档” → “支持用户上传并解析指定格式文档” |
| Q1 重要性 | 转成失败避免条件 | “避免漏掉任务” → “生成结果包含所有已确认任务项” |
| Q2 平台/性能 | 转成兼容或性能 AC | “默认 Web 应用” → “在目标浏览器环境完成主要流程” |
| Q3 技术栈 | 转成集成边界 | “使用现有数据库” → “不引入未确认的新数据库依赖” |
| Q4 不做事项 | 转成 direction_baseline，必要时也转成负向 AC | “不改认证模块” → “实现不修改认证模块行为” |

AC 编写规则：

- 每条以可验证动词开头：支持、返回、包含、通过、不抛出、不修改、拒绝、记录。
- 避免“更好”“完善”“智能”等不可验词。
- 每条只验证一个概念。
- 不足信息只能形成保守 AC，不得扩展范围。

## Step 2 — 方向基线推断

`direction_baseline` 用于每轮循环后核查方向一致性。提取顺序：

1. Q4 明确不做/不碰/不可降级内容。
2. Q2 的平台和环境边界。
3. Q3 的技术栈限制或“不新增未确认依赖”。
4. Q1 的目标边界，防止额外扩展。

默认方向基线仅在用户确认无约束时使用：

- “只实现已确认目标，不做目标外功能扩展”。
- “保持最小改动，不做无关重构”。
- “不得缩减已确认 AC 或跳过验证来换取交付”。

## Step 3 — 展示草案

锁定前必须向用户展示草案：

```markdown
以下是 PM 根据 Q1-Q4 推断的 Contract 基线草案，请确认是否锁定。

## Acceptance Criteria 草案
1. <AC 1>
2. <AC 2>

## Direction Baseline 草案
1. <baseline 1>
2. <baseline 2>

请回复“是”锁定，或逐条说明要修改的内容。
```

展示要求：

- 明确写“PM 推断”。
- 不写“根据文档事实”，除非 Q5 已升级 from-docs。
- 不把用户未确认的推断写入 Contract。

## Step 4 — 确认并锁定

锁定协议：

1. 用户明确确认“是”“确认”“锁定”“可以”或等价表达。
2. 将 AC 和 direction_baseline 写入 Contract 草案。
3. 标记这些条目为“用户确认基线”。
4. 后续 Phase 不得自行增加、删除、放宽或收紧这些条目。

Guard Clauses：

- 用户未确认 → 停止，不进入 Phase 3。
- 用户只确认部分条目 → 只锁定确认部分，剩余条目继续修改确认。
- 用户提出新约束 → 回到 Step 1/2 重新推断并展示。

## Step 5 — 变更流程

锁定后只有用户能修改基线。变更流程：

1. 用户明确要求修改某条 AC 或 direction_baseline。
2. PM 展示变更差异：旧条目 → 新条目。
3. 用户再次确认。
4. 更新 Contract，并记录“由用户变更确认”。

禁止行为：

- PM 因复杂度高自行删除 AC。
- task session 实施困难时放宽 direction_baseline。
- QA 发现难以验证时改写用户已锁定目标而不重新确认。
- 用“意思差不多”替代重新确认。

## 完成清单

| # | 检查项 | 必须满足 |
|---|--------|----------|
| 1 | AC 草案 | 每条可验证，且能追溯到 Q1-Q4 |
| 2 | direction_baseline | 与 Q4 一致，含默认最小改动边界或用户明确边界 |
| 3 | 展示 | 明确标注“PM 推断” |
| 4 | 确认 | 用户明确确认后才锁定 |
| 5 | 变更 | 锁定后修改必须重新确认 |

任一项缺失，禁止生成最终 Contract。
