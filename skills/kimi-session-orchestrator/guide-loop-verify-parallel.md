# 多 Session 并行验收

> 加载触发：Q2=B（规划派发与验收） + Q3=B（长轮次编排验收修复）。
> 将验收按维度拆分到独立 session，并行执行，交叉验证后聚合交付。

## §1 多维并行验收

按审查维度拆分，每个 session 负责一个独立维度：

| Session | 维度 | 关注点 |
|---------|------|--------|
| A | 正确性 | 逻辑、spec 一致性 |
| B | 完整性 | 需求覆盖率、遗漏 |
| C | 安全性 | 注入、权限、泄露 |
| D | 一致性 | 命名/格式/约定 |
| E | 性能 | 复杂度、I/O、内存 |

最少 2 个维度，最多 5 个——单维度失去并行意义。

## §2 派发流程

```
① create_session × N → ② execute_prompt × N（独立 criteria + grade_step）
③ Bash(run_in_background=true) × N — 并行后台轮询
④ 先完成先审查，不必等全部
```

**Prompt 要点**：指定维度、对照 spec、输出结构化问题清单（🔴🟡🟢 + 文件:行号）、只读不写。

## §3 交叉一致性检查

全部产出收集完毕后 PM 交叉验证：

```
逐维度审查 → 对比：同一文件上两个 session 的结论是否矛盾？
```

| 情况 | 处理 |
|------|------|
| 不同维度、不同结论 | 不矛盾，直接合并 |
| 同一事实、相反结论 | 矛盾 → 创建调解 session 裁决 |
| 调解 session prompt | "A 说 X，C 说 Y，请核实 `<文件:行>` 并裁决" |

## §4 聚合交付

```
① 合并所有维度清单 → ② 去重（同文件:行的同问题保留一次）
③ 按严重度排序 🔴→🟡→🟢 → ④ 输出报告
```

**报告结构**：总体结论 + 各维度发现（附维度标签）+ 矛盾项及调解结论 + 未覆盖项/风险。

## §5 并行上限

| 规则 | 原因 |
|------|------|
| 最多 5 个并行 session | 超过 5 → PM 审查超载、交叉对比 C(5,2)=10 对 |
| 同文件 ≤ 3 session 覆盖 | 冗余且矛盾概率激增 |

## 工具速查

**并行派发** `create_session` → `execute_prompt` → `Bash(poll_command)`
**交叉验证** `list_io_records` → 对比 → 必要时 `create_session` 调解
**聚合输出** `stream_response`（推送 WebSocket）
