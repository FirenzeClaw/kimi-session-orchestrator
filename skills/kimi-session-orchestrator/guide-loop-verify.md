# 验收循环 — PM 操作指南

> **加载触发**：PM 选择 Q2=B（verify loop）。聚焦输出验收、逐项打分、失败修复、重验放行。

---

## §1 验收循环模型

```
完成输出 → grade_step 逐项评分 → 失败清单 → 修复 → grade_step 重验 → 全绿 → 放行
```

每个工作包产出必须经过此循环才能交付。未通过验收的输出不进合成阶段。

---

## §2 grade_step 验收模式

将验收标准写成**具体、可度量**的检查项：

> "产出 MUST 包含章节标题 X, Y, Z"
> "src/types.ts 中的类型定义 MUST 与 spec §3.2 字段一一对应"

单次 grade 仅作参考；多次交叉验证提升置信度。

---

## §3 验收决策表

| 结果 | 决策 |
|------|------|
| **全部通过** | 进入合成/交付 |
| **部分失败** | 生成失败清单 → 派发 fix session 或 PM 手动修复 |
| **重验** | 同一标准、同一 grade_step，修复后重新打分 |

---

## §4 验收标准编写指南

每条标准必须**独立可验**——不需要额外上下文即可判断通过/失败。

| ❌ 差 | ✅ 好 |
|------|------|
| "代码没问题" | "src/types.ts 中所有 TypeScript 类型与 spec §3.2 字段定义一致" |
| "功能正确" | "POST /api/submit 返回 201 Created，且 body 符合 contract §5.1 schema" |
| "文档完整" | "AGENTS.md 存在 §修改记录 条目，包含日期、作者、变更摘要" |
